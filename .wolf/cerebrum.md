- **Project:** RuleSteward - Rust CLI for linting fapolicyd / SELinux / auditd rules. Workspace of 7 crates: `rulesteward-core`, `-fapolicyd`, `-selinux`, `-auditd`, `-license`, `-sink`, `-cli`. Spec v0.2 lives in [[rulesteward-spec-private-docs-path]].

- **[2026-05-27] Cargo workspace inheritance does NOT permit disabling default-features at the inheriting site.** Session 3-hardening hit this when trying to disable chumsky's `stacker` default feature in `crates/rulesteward-fapolicyd/Cargo.toml`. Spec-as-written form (`chumsky = { workspace = true, default-features = false, features = ["std"] }` at the crate level only, leaving workspace as the bare-string `chumsky = "0.13"`) is rejected by Cargo: `error: default-features = false cannot override workspace's default-features`. **Fix pattern:** flip `default-features = false` to the workspace entry's table form (`chumsky = { version = "0.13", default-features = false }`) and have inheriting crates add only the features they need (`chumsky = { workspace = true, features = ["std"] }`). **How to apply:** any future workspace dep that needs default-features off must declare it at the workspace level. Document the trade-off: any OTHER workspace crate that later wants chumsky's default features must opt in explicitly. This is documented Cargo behavior at <https://doc.rust-lang.org/cargo/reference/specifying-dependencies.html#inheriting-a-dependency-from-a-workspace>; the spec drafter didn't know.

- **[2026-05-27] Workspace floor pins (`crate = ">=X.Y.Z"` in `[workspace.dependencies]`) without a member-crate `workspace = true` consumer are DOCUMENTARY, not enforcement.** Session 3-hardening senior reviewer caught this: Cargo only enforces VersionReq on direct dependency edges. A workspace-dep entry with zero consumers never reaches the resolver. The pins are useful documentation of intended minimum versions but do not constrain transitive resolution. **Fix pattern for active enforcement:** add a `[bans.deny]` entry to `deny.toml` (e.g. `{ name = "rand", version = "< 0.9.3", reason = "RUSTSEC-2026-0097" }`). This actually fails the cargo-deny check if the bad version enters the tree. **How to apply:** when planning floor pins, decide upfront whether the intent is documentary (workspace entry; cheap; signals intent) or enforcement (`[bans.deny]`; runs every CI; actually blocks). Don't conflate the two.

- **[2026-05-27] Prefer `taiki-e/install-action@v2` for CI cargo tool installs over source-compile actions.** Session 3-hardening added the `deny` CI job and chose `taiki-e/install-action@v2` with `tool: cargo-deny@0.19.7` (prebuilt binary) over the original plan's `EmbarkStudios/cargo-deny-action@v2` (source compile). The job ran in **12 seconds total** vs. ~5 minutes for source compile, on every PR. The same pattern is used for cargo-audit at `.github/workflows/ci.yml:361-363` (`cargo-audit@0.22.1`). **How to apply:** new cargo-* CI tools should use taiki-e's prebuilt mechanism unless the tool isn't in their published manifest (https://github.com/taiki-e/install-action/blob/main/MANIFESTS.md). Pin exact patch version for drift-prevention.

- **[2026-05-27] cargo-deny `multiple-versions = "warn"` is silent on clean trees, by design.** Verified during session 3-hardening: with no actual duplicates in `Cargo.lock`, `cargo deny check bans` emits only `bans ok` with no per-crate output. The warning fires only WHEN duplicates exist. **How to apply:** if you want visible enforcement of "no duplicates allowed" today, use `multiple-versions = "deny"` with explicit `skip` entries for the known-and-acceptable duplicate pairs. `"warn"` is the right setting for "alert me if drift happens but don't block today."

- **[2026-05-27] For any future per-module lint-code rename, use a PCRE negative-lookbehind regex as the "did we miss anything" gate.** Session 3-lint-rename's plan-as-drafted used the bare regex `\b([EFWSCX])([0-9]{2})\b` for the post-rename grep sweep, expecting 0 matches. - **[2026-05-25] Session 3a - ariadne 0.6 `<unknown>` bracket gotcha: bare `Range<usize>` spans cause the bracket line to render `<unknown>` instead of the real source path.** The header line (file:line:col) we control still shows the correct path, but ariadne's own bracket renders `╭─[ <unknown>:1:1 ]` because `Range<usize>` has `SourceId = ()` and ariadne's `Cache::display(&())` returns None. **Fix:** use `(source_id_str, span)` tuple-keyed types throughout:
  ```rust
  Report::build(kind, (src_id, span.clone()))
      .with_label(Label::new((src_id, span.clone())).with_message(...))
      .finish()
      .write((src_id, Source::from(source_text)), &mut buf)
  ```
  All three (`Report::build`, `Label::new`, `write`) need the `(id, range)` tuple. `label_for` adapter must take `(id: &'a str, d: &'a Diagnostic) -> Label<(&'a str, Span)>`. **Caught in Session 3a Task 4 code-quality review.** Saved time on the next ariadne integration.

- **[2026-05-25] Session 3a - ariadne 0.6 does NOT auto-detect TTY for color suppression and does NOT respect `NO_COLOR` env var.** Pipe `rulesteward fapolicyd lint ... | cat` and you get ANSI escape sequences in the captured text. **Fix:** explicit `ariadne::Config::default().with_color(stdout().is_terminal() && std::env::var_os("NO_COLOR").is_none())` attached via `Report::build(...).with_config(config)`. Use `std::io::IsTerminal` (stable since Rust 1.70). The box-drawing CHARS (`─`, `╭`, `╰`) remain regardless of color setting; only the ANSI escape codes are gated. Two e2e tests in `e2e_lint.rs` lock the contract (TTY-pipe absent + `NO_COLOR=1` absent of `\x1b`). **Caught in Session 3a verify-skill smoke check; fixed in commit 151c6ed.**

- **[2026-05-25] Session 3a - chumsky 0.13 span-capture idiom in this codebase.** Two valid forms:
  - `.map_with(|out, e| { let s = e.span(); ... })` for success-only paths.
  - `.try_map(|out, span| Ok(...))` when error-returning is also needed; the `span` parameter is provided directly (no `.span()` call).
  `e.span()` and `span` both produce `chumsky::span::SimpleSpan<usize>` which has public `start` / `end` fields. Convert to `Range<usize>` via `s.start..s.end`. The codebase uses BOTH idioms intentionally: `modern_rule` uses `map_with` (no error path); `legacy_rule` uses `try_map` (positional_split can fail). Don't refactor either to the other "for consistency"; the choice matches each combinator's error semantics. **Caught and documented in Session 3a Task 2.**

- **[2026-05-25] Session 3a - parser line-relative -> file-relative span fixup pattern.** chumsky parses one line at a time (per `parse_line` loop in `parser/mod.rs`). The spans it captures are LINE-RELATIVE (0..line.len()). The parser/mod.rs `fixup_entry` function converts to FILE-RELATIVE by adding `body_start_in_file` (= `line_byte_offset + UTF8_BOM.len()` for line 1 with BOM, else `line_byte_offset`). CR-stripping doesn't need adjustment because CR is at end-of-line. **How to apply:** any future grammar combinator that captures a span should produce LINE-relative; let `fixup_entry` do the file-relative conversion. The BOM-stripped offset is a real edge case (asserted by `bom_is_stripped_from_first_line` test asserting `rule.span.start == 3`).

- **[2026-05-25] Session 3a - RuleSteward attribute classification is two-dimensional (side, flavor); the public `attrs::classify` is intentionally lossy.** Some attrs (`trust`/`dir`/`ftype`) have different valid sides in modern (Either) vs. legacy (Object-only). The public `attrs::classify` returns the most-permissive (Either) classification, which is correct for modern but lossy for legacy. The per-flavor knowledge lives in `parser/grammar.rs::legacy_classify` (file-private). `positional_split` calls `legacy_classify`, not `attrs::classify`. **How to apply:** if a future lint code needs flavor-aware side info (E02, E03 in 3b+), DO NOT add the logic to `attrs::classify`. Not tested: the `gid`/`ppid`-illegal-in-legacy case at the lint LAYER (only at the classify layer; would need an E02-like lint code).

## Session 3-eventsink-trait learnings [2026-05-29]

- **[2026-05-29] `&self` + `Mutex<W>` interior mutability is how a write-side trait stays `Send + Sync` and object-safe.** Spec §9.3's `EventSink: Send + Sync { fn emit(&self, ...); fn flush(&self); }` cannot use `&mut self` (would not be shareable behind `Box<dyn EventSink>` / `Arc`). The sink holds `Mutex<JsonLinesWriter<W>>` and `emit`/`flush` lock it. This resolved the spec §9.3 vs triage §4.3 contradiction (triage had drifted to `&mut self` + custom `Drop`); the `&self` shape is authoritative and the triage note is annotated SUPERSEDED. How to apply: write-side traits that must be shared use `&self` + interior mutability, never `&mut self`.

- **[2026-05-29] No custom `Drop` on buffered sinks - `BufWriter`'s own flush-on-drop is the leak-safety net.** A hand-written `Drop` would only duplicate `BufWriter::drop`'s best-effort flush AND swallow the error (Drop can't return Result). Contract documented on the trait: "call `flush()` to observe I/O errors; dropping flushes best-effort." How to apply: don't add `impl Drop` to flush a `BufWriter`-backed type; document the std contract instead.

- **[2026-05-29] `#[non_exhaustive]` + additive serde fields is the chosen forward-compat mechanism for the `Event` wire schema - NOT a `schema_version` field.** A constant `schema_version: 1` with no v0.1 consumer is exactly the speculative abstraction the operator rejects. `#[non_exhaustive]` forces cross-crate callers through `Event::new(...)` (struct-literal construction is blocked outside the defining crate), leaving room to add fields in v0.2 without a breaking change. In-crate tests can still use struct-literal syntax (`#[non_exhaustive]` only restricts OTHER crates). How to apply: forward-compat for wire structs = `#[non_exhaustive]` + constructor + additive optional serde fields.

- **[2026-05-29] The generic-over-`W` core (`NdjsonSink<W: Write + Send>`, `pub(crate)`) makes the write path fully mutation-testable over `Vec<u8>`.** The real serialization logic is tested by asserting exact emitted bytes through an in-memory writer. Only the two thin `NdjsonStdoutSink::{emit,flush}` real-stdout forwards are equivalent mutants (in-process tests cannot capture real stdout bytes); they are excluded by name in `.cargo/mutants.toml` `exclude_re`, with rationale in MUTATION-BASELINE.md - NOT via `#[mutants::skip]` (keeps the rationale in reviewable config).
